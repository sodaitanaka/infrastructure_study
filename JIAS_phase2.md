**プロジェクト名:** JIAS様 CDP（カスタマーデータプラットフォーム）構築プロジェクト
**作成日:** 2026年3月26日
**バージョン:** Ver 1.0
**作成者:** 株式会社Flued

## 1. プロジェクト概要

### 1.1 Phase2の背景・目的

Phase1で構築したDWHは「社内の経営・営業分析基盤」として機能している。Phase2では、このDWHを起点に **顧客向けCDP（カスタマーデータプラットフォーム）** へと拡張する。

基幹システム（Shopify経由）に蓄積された購買データと、HubSpotで管理するCRM情報を統合することで、以下の2つのゴールを実現する。

| # | ゴール | 詳細 |
|---|--------|------|
| 1 | **顧客マイページの実現** | HubSpot上にログイン機能付きのマイページを構築し、各顧客が自身の購買履歴（いつ・何を買ったか）を閲覧できるようにする |
| 2 | **購買データのマーケ活用** | 購買履歴・LTV・セグメント情報をHubSpotに連携し、パーソナライズされたマーケティング施策（メール配信・リスト作成等）に活用する |

### 1.2 Phase1との対比

| 項目 | Phase 1（完了） | Phase 2（今回） |
|------|-----------------|-----------------|
| データソース | 基幹システム（Shopify経由） | 基幹システム（Shopify経由）＋ **HubSpot CRM** |
| DWH | BigQuery（Raw/Intermediate/Mart） | BigQuery（拡張）＋ **CDP Mart追加** |
| 出力先 | LookerStudio | LookerStudio ＋ **HubSpot（逆連携）** |
| 主な利用者 | 経営層・営業担当（社内） | ＋ **顧客（エンドユーザー）** |
| 主目的 | 社内分析・Salesforce脱却 | ＋ **顧客体験向上・マーケティング自動化** |

### 2.4 HubSpot連携方式の詳細

#### A. HubSpot → DWH（データ取込）
- **方式:** FiveTran の HubSpot コネクタ（ネイティブ対応）
- **対象オブジェクト:** コンタクト、会社（Companies）、取引（Deals）、エンゲージメント（メール開封・クリック等）
- **更新頻度:** 日次（既存の基幹データ取込に合わせる）

#### B. DWH → HubSpot（逆連携：Reverse ETL）
- **方式:** Hightouch を使い BigQuery のMartテーブルをHubSpotへ同期
- **同期対象:** 累計購買金額、購買回数、最終購買日、商品カテゴリ、顧客セグメント（RFM）
- **同期先:** HubSpot コンタクトのカスタムプロパティ
- **更新頻度:** 日次（夜間バッチ）
- **代替案:** Google Cloud Functions + HubSpot API（追加コストを抑えたい場合）

#### C. 顧客IDマッチング
- **課題:** 基幹システムの「得意先コード」とHubSpotの「Contact ID」は別体系
- **解決方針:** メールアドレスを共通キーとして突合。Intermediate層に `int_customer_id_mapping` テーブルを作成し、両システムのIDを管理
- **名寄せロジック:** メール完全一致 → 電話番号一致 → 会社名＋担当者名一致（優先順）

### 3.4 非機能要件

| 項目 | 要件 |
|------|------|
| データ鮮度 | 購買データのHubSpot反映は翌日（日次バッチ） |
| マイページ応答速度 | ページ表示3秒以内 |
| セキュリティ | HubSpotのSSO・ポータル認証を利用（パスワードはHubSpot管理） |

| 障害対応 | 同期失敗時はSlackアラート通知・翌日リトライ |

## 4. DWHに追加が必要なMart

### 4.1 新規Raw層テーブル（HubSpot）

| テーブル名 | 元データ | 主な内容 |
|-----------|---------|---------|
| `raw_hubspot_contacts` | HubSpot コンタクト | 氏名、メール、電話、属性情報 |
| `raw_hubspot_companies` | HubSpot 会社 | 会社名、業種、担当者 |
| `raw_hubspot_deals` | HubSpot 取引 | 商談情報（必要に応じて） |
| `raw_hubspot_email_events` | HubSpot メールイベント | 開封・クリック・配信停止 |

### 4.2 Intermediate層の拡張

| テーブル名 | 役割 |
|-----------|------|
| `int_customer_id_mapping` | 基幹顧客コード ↔ HubSpot Contact IDのマッピング管理 |
| `int_unified_customers` | 基幹顧客マスタ ＋ HubSpotコンタクトを結合した統合顧客テーブル |
| `int_customer_purchases` | 顧客別購買明細（マイページ・Mart向けの中間テーブル） |

### 4.3 新規Mart定義

#### M6: CDP統合顧客マート

| 項目 | 内容 |
|------|------|
| **目的** | 基幹顧客情報とHubSpot CRM情報を1顧客1レコードで統合管理 |
| **主要カラム** | unified_customer_id, 得意先コード, hubspot_contact_id, 氏名, メール, 会社名, 累計購買金額, 購買回数, 最終購買日, 顧客ステータス |
| **用途** | Reverse ETLの基準テーブル、マイページ認証基盤、他Martの親テーブル |
| **更新頻度** | 日次 |
| **データソース** | raw_customers, raw_hubspot_contacts, int_customer_id_mapping |

#### M7: 顧客別購買履歴マート

| 項目 | 内容 |
|------|------|
| **目的** | マイページで表示する個人別購買履歴を格納。HubSpotに逆連携する購買サマリーの計算元 |
| **主要カラム** | unified_customer_id, 購買日, 商品コード, 商品名, カテゴリ, 数量, 金額, 税区分, 売上明細番号 |
| **用途** | HubSpotカスタマーポータルのマイページ表示（Reverse ETL経由） |
| **更新頻度** | 日次 |
| **データソース** | raw_sales_details, int_customer_purchases, M6 |

#### M8: 顧客セグメント・LTVマート

| 項目 | 内容 |
|------|------|
| **目的** | RFM分析によるセグメント分類とLTVスコアを計算。マーケ施策のターゲットリスト作成に使用 |
| **主要カラム** | unified_customer_id, recency_days（最終購買からの日数）, frequency（購買回数）, monetary（累計購買金額）, rfm_segment（例：VIP/アクティブ/休眠/離脱）, ltv_score, 最終購買日, 初回購買日 |
| **用途** | HubSpotセグメントリスト、メール配信ターゲット、LookerStudio CDP分析ダッシュボード |
| **更新頻度** | 日次 |
| **データソース** | M6, M7 |

### 4.4 マート追加サマリー

| マートNo | マート名 | Phase1との関係 | 主な出力先 |
|---------|---------|---------------|-----------|
| M1〜M5 | 既存マート | 変更なし（継続運用） | LookerStudio |
| **M6** | CDP統合顧客マート | 新規（HubSpot統合） | Hightouch → HubSpot |
| **M7** | 顧客別購買履歴マート | 新規（マイページ用） | Hightouch → HubSpot |
| **M8** | 顧客セグメント・LTVマート | 新規（マーケ活用） | Hightouch → HubSpot、LookerStudio |

| 2 | HubSpot カスタマーポータル利用のためService Hub Professionalが契約されていること |
| 3 | 基幹システムの顧客データとHubSpotコンタクトにメールアドレス等の共通キーが存在すること |
| 4 | Shopify購買データが基幹システム経由でDWHに連携済みであること（Phase1の延長） |
| 5 | Phase1のDWH基盤（BigQuery/FiveTran/dbt）が安定稼働していること |

#### リスクと対策

| リスク | 影響度 | 対策 |
|--------|-------|------|
| 顧客IDマッチング率が低い（共通キーが存在しない顧客が多い） | 高 | 設計フェーズでサンプルデータで先行確認。手動マッチング用の管理画面検討 |
| HubSpotプランが不足（カスタマーポータル・Ops Hub非契約） | 高 | 要件定義前にHubSpotプランを確認・必要に応じてアップグレード見積もり |
| Reverse ETLツール（Hightouch）のコスト増 | 中 | 同期対象テーブル数を絞ってコスト最適化。代替としてCloud Functions検討 |
| 購買履歴データ量が多くHubSpotプロパティに全件格納できない | 中 | 直近1〜2年分に絞って同期。全履歴はマイページへのリンクで対応 |
| マイページのUI表示にHubSpot CMS Hubが必要になるケース | 低 | カスタマーポータルの機能範囲を事前確認。必要ならCMS Hub追加を検討 |
