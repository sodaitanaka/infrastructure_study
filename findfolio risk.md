# エラー監視方式の検討：Sentry vs メール通知

**対象システム:** FINDFOLIO与信管理システム
**作成日:** 2026年3月26日
**背景:** 現行仕様ではSentry統合が設計されているが、findfolio newsチェックで実績のあるメール通知（自前実装）で代替可能かを検討する。

---

## 1. 現行設計（Sentry統合）の概要

overview.mdのアーキテクチャ図では、Firebase Functions からエラーをSentryに送信する構成が定義されている。

```
Firebase Functions --[エラー送信]--> Sentry（エラー監視）
```

---

## 2. 比較：Sentry vs メール通知

### 2.1 機能比較

| 観点 | Sentry | メール通知（自前実装） |
|------|--------|----------------------|
| **スタックトレース取得** | 自動で詳細なトレースを記録 | メール本文に手動で含める必要あり |
| **エラーの集約・重複排除** | 同一エラーをグループ化して件数管理 | 都度メールが届く（大量発生時に埋もれる） |
| **エラーの検索・絞り込み** | ダッシュボードで検索・フィルタ可能 | メールボックス検索のみ |
| **アラートの柔軟な設定** | 発生頻度・特定エラー種別での通知制御 | 全エラーで同じ通知（制御が困難） |
| **デプロイとの紐付け** | リリースバージョンごとにエラー分類可能 | なし |
| **エラー解決ワークフロー** | resolved / ignored などのステータス管理 | なし |
| **複数エンドユーザー識別** | context にテナント情報を付与して分類 | メール本文で手動識別 |

### 2.2 コスト比較

| 観点 | Sentry | メール通知（自前実装） |
|------|--------|----------------------|
| **外部サービス依存** | あり（Sentry SaaS） | なし |
| **月額費用** | Developerプラン: 無料（イベント5,000件/月）<br>Teamプラン: $26〜/月（無制限） | なし（メール送信コストのみ） |
| **実装コスト** | SDK導入のみ（低い） | カスタム関数の実装が必要（findfolio newsの実績あり） |
| **運用負荷** | Sentryのダッシュボード管理が必要 | メールボックス管理のみ |

### 2.3 本システム特有の考慮点

| 観点 | 評価 |
|------|------|
| **エラー発生頻度の想定** | RM APIのタイムアウト・認証エラーが主な発生源。高頻度ではないが、複数テナント分が重なる可能性あり |
| **エラー種別の多様性** | `parameter_error` / `permission_denied` / `network_timeout` など複数種別が存在し、分類して対応が必要 |
| **マルチテナント構成** | エンドユーザーごとのAPIキーを使用するため、どのテナントで発生したか特定できることが重要 |
| **Cloud Tasksによる再試行** | ネットワーク系エラーはHTTP 500返却→Cloud Tasksが自動再試行するため、一時的エラーは自然解消する設計 |
| **findfolio newsでの実績** | メール通知（`send_system_error()`）で本番運用の実績あり |

## 3. エラー種別ごとの対応必要度

本システムで想定されるエラーと、各方式での対応力を整理する。

| エラー種別 | 発生源 | 緊急度 | Sentry | メール通知 |
|-----------|--------|--------|--------|-----------|
| RM API タイムアウト | RM API | 低（再試行で解消） | ◎ 自動集約 | ○ メール通知（大量発生時は埋もれる） |
| RM API 認証エラー（APIキー不正） | Airtable/RM API | 高（テナント特定が必要） | ◎ context付きで即時特定 | △ メール本文で確認 |
| HubSpot API エラー | HubSpot Webhook | 中 | ◎ | ○ |
| Airtable 接続エラー | Airtable | 高（全テナント影響） | ◎ | ○ |
| 未補足の例外（バグ） | Firebase Functions | 高（修正要） | ◎ スタックトレース自動取得 | △ 手動でトレース記述が必要 |
| 上限超過アラート | 利用量集計 | 中 | △（本来の用途外） | ◎ 専用関数で対応 |

## 4. 結論・推奨

### 4.1 Sentryなしで運用できるか

**→ 技術的には可能。findfolio newsで実証済み。**

ただし以下の条件が揃っている場合に限り許容範囲と判断できる：

- エラー発生時にテナント情報（会社名・HubSpotポータルID等）をメール本文に含める実装をする
- スタックトレースをメール本文に含める実装をする
- 上限超過・上限近接の通知は別関数（`send_alert_over_limit()` 等）で対応する
- Cloud Tasksの再試行ログはGCP Cloud Loggingで確認できる体制を整える

### 4.2 Sentryを残すべきケース

以下のいずれかに該当する場合はSentryの維持を推奨：

- リリース初期でバグ発生率が読めない段階
- テナント数が増加してエラーの分類・特定コストが高くなった時
- 同一エラーが大量発生してメールが埋もれるリスクが高い時

### 4.3 段階的移行案

初期はメール通知のみで運用し、以下のタイミングでSentryの導入を検討する方式も有効：

```
フェーズ1（初期）: メール通知のみ（findfolio newsと同方式）
        ↓
フェーズ2（テナント数増加後）: Sentry導入を再検討
```

## 5. メール通知で代替する場合の実装要件

Sentryなしで運用する場合、`send_system_error()` に以下の情報を含めることを必須とする：

```python
# 必須含有情報
- エラー種別（Exception type）
- スタックトレース（traceback.format_exc()）
- テナント識別情報（HubSpotポータルID / 会社名）
- 発生日時
- 実行関数名・処理ステップ

# 例
send_system_error(
    error_type="RMAPIAuthError",
    traceback=traceback.format_exc(),
    tenant_id=portal_id,
    company_name=company_name,
    step="rm_api_credit_check"
)
```

通知先は環境変数 `EMAIL_ALERT_TO` で管理（findfolio newsと同方式）。
